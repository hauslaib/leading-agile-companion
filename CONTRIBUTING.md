# Contributing

Thanks for considering a contribution. This repository collects facilitation materials for agile leaders working through disagreement. Contributions that fit that frame are welcome from practitioners at any level.

## What's in scope

- New scenarios drawn from real engagements (anonymised — see below).
- Alternate retrospective formats, particularly ones suited to teams in conflict or misalignment.
- Worksheets and canvases that complement the existing set.
- Translations, especially German. Other languages are welcome too.
- Corrections, clarifications, and improvements to existing materials.

## What's out of scope

- The manuscript of *Leading Agile When No One Agrees* itself. This repository is the companion, not the book.
- Generic agile or SAFe content already well covered elsewhere (e.g., standard Scrum ceremonies, basic PI planning explainers).
- Promotional material, tool advertisements, or vendor-specific content.

## Anonymisation rule

Every scenario must be anonymised before it is published here. That means:

- No real company names, product names, or project codenames.
- No identifying details about individuals — change roles, team sizes, geographies, and timelines where needed to obscure the source.
- Composite scenarios drawn from multiple engagements are encouraged and often stronger than single-source ones.
- If you are unsure whether a detail is identifying, change it.

This applies to your own past engagements as much as anyone else's. Confidentiality obligations do not lapse because a project ended.

## How to contribute

1. Open an issue first for anything substantial (a new scenario, a new worksheet, a new retrospective format). A short proposal — the situation, the tension, what the artifact teaches — is enough. This avoids duplicate work and lets us discuss scope before you write.
2. Fork the repository and create a branch named after the contribution, e.g. `scenario/architecture-vs-product` or `worksheet/decision-log`.
3. Write in clear, plain English (or German). Short sentences. Concrete examples. No filler.
4. Match the structure of existing files in the same folder. Each folder's `README.md` describes the format expected.
5. Open a pull request referencing the issue. One contribution per PR keeps review tractable.

## Style notes

- Markdown only. No HTML unless strictly necessary.
- Headings sentence case, not Title Case.
- Avoid words like "critical" and "crucial". Avoid hype generally.
- Worksheets should be usable as-is by someone facilitating tomorrow morning. If a section needs lengthy explanation to use, the worksheet is not done.
- Examples beat descriptions. A filled-in version of any new worksheet should accompany the blank template.

## Translations

Translations live alongside the original file with a language suffix, e.g. `conflict-map.de.md` next to `conflict-map.md`. Keep filenames and section anchors consistent across translations so cross-references work.
